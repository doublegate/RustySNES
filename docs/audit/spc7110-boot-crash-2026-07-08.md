# SPC7110 boot-crash investigation

**Status: root cause #2 (the boot-blocking one) is CLOSED, and it was never a RustySNES bug.**
The local ROM dump this whole investigation traced against
(`tests/roms/external/commercial/HiRom/SPC7110/Tengai Makyou Zero.sfc`, 7 MiB) is the **English
fan-translation ROM hack of this game, not the original Japanese cartridge** — confirmed
independently three ways (SHA256 mismatch against the authoritative database, a header-checksum
inconsistency, and a public forum post describing this exact patch's custom memory map, all
below). The translation patch adds a 1 MiB "Expansion ROM" region at CPU banks `$40-$4F` that
does not exist on any real cartridge and that RustySNES — correctly, per the real board's own
database entry — does not map. The literal `JSL $4FFB80` this investigation traced down to is
the patch jumping into its own expansion code; on a genuine original-cartridge dump, this
instruction (or the bytes around it) would not be present at all. See "The resolution" below for
the full evidence chain. Root cause #1 (`bus_mirror` addressing, `v0.4.0`) and the four
independently real bugs found and fixed along the way in `v0.8.0` (DCU/ALU trigger timing; the
`$40-$7D` mcu-mapping fix; the DROM-buffer oversizing fix; the systemic cart-layer open-bus fix)
all stand as genuine accuracy improvements — this investigation's tooling and fixes were sound
throughout; only the ROM dump used to drive it was the wrong artifact.

**Board:** `crates/rustysnes-cart/src/coproc/spc7110.rs` (`coproc::spc7110`), `BestEffort` tier.
**Test title:** Far East of Eden Zero (Tengai Makyou Zero) — the one commercial SPC7110 title
with a local dump in this project's corpus. No other SPC7110 title has been tried.
**Spec entry:** `docs/cart.md`'s SPC7110 bullet (kept as the current-state summary; this file
is the full trail behind it).

## Background

SPC7110 is a Hudson ASIC combining a decompression unit (an adaptive binary range coder over
1/2/4bpp planes), a data-port unit, an ALU, and a memory-control unit that exposes four
independently-bankable 1 MiB data-ROM windows (`$4830`-`$4833` select, `$4834` selects the
window granularity). Unlike every other coprocessor in this project, SPC7110 carts physically
carry **two separate ROM chips** — a small PROM (CPU-executable program code, LoROM-style banks
`$00`-`$0F`) and a much larger DROM (compressed/plain data, addressed only through the board's
registers) — concatenated into one raw `.sfc` dump. There is no header field or generic formula
that recovers the PROM/DROM split for an arbitrary SPC7110 title; `coproc::spc7110::select`
guesses it from Far East of Eden Zero's documented physical geometry (1 MiB PROM + the
remainder as DROM, when the image is at least 2 MiB — see `select`'s doc comment for the
"no PROM" fallback used by titles like Momotarou Dentetsu VII, untested here).

## Root cause #1 — `bus_mirror` addressing (confirmed, fixed in `v0.4.0`)

### Symptom

Booting Far East of Eden Zero: the CPU ran off into unmapped low banks roughly 20-30 frames
into boot, BRK-storming (repeatedly trapping into the BRK vector, never recovering) per the
original diagnostic trace.

### Root cause

`datarom_read`/`mcurom_read`'s PROM/DROM lookups used a plain `offset % len` fold to keep a
register-selected address inside the physical chip's buffer. Real hardware (ares' `Bus::mirror`,
`sfc/memory/inline.hpp`) does something different: it repeatedly strips the largest power-of-two
block that keeps the address in range. The two algorithms agree only when the buffer size is
itself a power of two — and Far East of Eden Zero's DROM is not: `select` splits a 7 MiB image
into a 1 MiB PROM and a **6 MiB** DROM, and 6 MiB is not a power of two.

`$4830`-`$4833` can select up to an 8 MiB addressable window, larger than the physical 6 MiB
DROM. A register-selected read past the physical chip size but inside that addressable window
hit the `% len` fold and returned the **wrong byte** — silently corrupting whatever
data-ROM-resident table the game was reading through the data-port unit at the time.

### Fix

Ported the real `Bus::mirror` algorithm as `spc7110::bus_mirror` (`spc7110.rs:757`,
`const fn bus_mirror(address: u32, size: u32) -> u32`) and applied it to every PROM/DROM lookup
in `datarom_read` (`spc7110.rs:191`), `datarom_read_static` (`spc7110.rs:321`, the const-context
sibling used by the const-evaluable read paths), and `mcurom_read` (`spc7110.rs:432`).

### Effect (measured, not assumed)

The wild-PC excursion moved from ~20-30 frames into boot (permanent BRK-storm crash) to ~90+
frames, and the CPU now **self-recovers** via a BRK/RTI oscillation instead of crashing
permanently. This is real, measured forward progress — not a full fix — hence root cause #2
below.

## Root cause #2 — WRAM never populated before a real `RTI` targets it (open)

### Symptom (current, as of `v0.4.0`)

The CPU eventually executes a genuine `RTI` instruction, traced to `$CD:D2B3` — confirmed to be
real PROM-resident code, not decompression garbage or a wild-PC artifact. That `RTI` pops a
return address off the stack pointing into WRAM: `$20:0848` (mirror of `$7E:0848`). That WRAM
location is confirmed **entirely zeroed** at the point the `RTI` executes. The CPU then starts
"executing" zeroed WRAM as if it were code (`BRK` is opcode `$00`), traps into the BRK vector,
and the whole sequence oscillates — this is the "~90+ frames, self-recovering BRK/RTI
oscillation" behavior root cause #1's fix produced, not a full boot.

### Working hypothesis (not yet confirmed)

The game expects real code to already be resident at `$7E:0848` by the time this `RTI` fires —
very likely written there via a DCU-driven (decompression-unit-driven) decompress-to-WRAM step
earlier in the boot sequence that either (a) hasn't run yet when this `RTI` fires (an ordering
bug — this board's `coprocessor_tick`/data-port timing runs too slow or too fast relative to the
CPU's own progress), or (b) has run but is itself still buggy (a second, distinct data-path bug
in the decompression unit or the data-port write path that silently produces zeroed/wrong output
instead of the real decompressed code). Both are plausible from this board's implementation
maturity (`BestEffort` tier, no other data-path bug has been independently confirmed yet); this
has **not** been narrowed further than "one of these two."

### `v0.8.0` update — DCU/ALU trigger timing fixed (real bug, independently confirmed), but not
this one

Reading `ref-proj/ares`'s `sfc/coprocessor/spc7110/{dcu.cpp,spc7110.cpp,spc7110.hpp}` directly
confirmed real hardware's SPC7110 runs as its own cothread (`Thread::create(21'477'272, ...)`,
exactly the master-clock rate) whose `main()` **defers** a `$4806` DCU-begin-transfer / `$4825`
multiply / `$4827` divide trigger by one thread tick (`dcuPending`/`mulPending`/`divPending`),
rather than completing it synchronously within the triggering write — this project's board had
never modeled that deferral. Ported faithfully: `Spc7110Board` gained `dcu_pending`/
`mul_pending`/`div_pending` flags set by the register write and consumed one master-clock tick
later in a new `Board::coprocessor_tick` override (`spc7110.rs`, matching the same
per-master-clock hook GSU/SA-1 already use for their own host-synced timing). 9/9 unit tests
pass, including a new one proving the one-tick deferral directly. This is a genuine,
independently-verified timing-accuracy fix, not a guess — but it does **not** fix root cause #2
(confirmed below), because the DCU/ALU triggers this fix affects are never actually written
during this boot's crash path in the first place.

### `v0.8.0` update — watchpoint-based tracing substantially narrows root cause #2

Using the new `Bus::set_watchpoints`/`take_watchpoint_hits` hook (`T-81-001b`,
`rustysnes-core::watchpoint`) directly against `System::bus` in a throwaway diagnostic test
(booting the real Far East of Eden Zero dump headlessly, watching `$7E0800-08FF` writes, the
`$4806`/`$4825`/`$4827` triggers, every SPC7110 register `$4800-483F` + RTC `$4840-4842` read,
`$4200`/`$4210` NMI enable/ack, `$2140-2143` APU I/O ports, and `$4212` HVBJOY), the following is
now **measured, not hypothesized**:

- `$7E0800-08FF` (which contains the crashing `RTI`'s `$0848` target) is written **exactly once**,
  at reset (`pc=$00F345`, a bulk WRAM-clear loop, all zero) — and never again, across 3600 frames
  (60 real seconds) of boot.
- `$4806`/`$4825`/`$4827` (the DCU/ALU triggers the `v0.8.0` timing fix above targets) are
  **never written at all**. No SPC7110 register in `$4800-483F`/`$4840-4842` is ever **read**
  either. The DCU is never invoked in the first place — this **rules out both original
  hypotheses** ("hasn't run yet" and "ran but produced wrong output"): it simply never runs.
- By ~frame 20 the CPU settles into a stable two-PC loop — `$00:F416` (PROM) alternating with a
  drifting PC inside `$20:2086-20FA` (a WRAM bank, `$20:2000` mirrors `$7E:2000`) — that persists
  unchanged for the full 3600-frame trace. This is **not** the BRK/RTI-storm behavior
  root cause #1's original fix produced; it is a quieter, stable loop, suggesting `v0.8.0`'s
  scheduler-ordering fixes (the mid-scanline HDMA `dot_ticked` correction, `PR#60`) changed this
  board's exact crash shape even though nothing SPC7110-specific was touched by that PR.
- Holding the Start button continuously from frame 60 through frame 3600 (`Bus::set_joypad`)
  produced a **byte-identical** trace — ruling out "this is a normal, legitimate wait-for-start
  screen the headless test just never presses."
- `$4200` (NMITIMEN) is never written — NMI is never even enabled — and `$4210` (RDNMI) is never
  read. `$2140-2143` (the SPC700 IPL-ROM handshake ports) see zero traffic. `$4212` (HVBJOY) is
  never read. The loop is a pure CPU-internal spin touching **no bus-visible hardware state at
  all** beyond the reset-time WRAM clear.

**Conclusion:** root cause #2 is not a DCU/ALU trigger-timing bug (ruled out directly) and not a
"real code hasn't been written yet, wait a bit longer" situation (ruled out by the 60-second
trace). The CPU reaches a stable loop bouncing between PROM and a WRAM bank (`$20:20xx`) that was
**never written by anything after reset** — i.e., the same disease as the original `$0848`
symptom (executing never-populated WRAM as code), just relocated to a different address by
`v0.8.0`'s scheduler-ordering changes. The bug is upstream of any SPC7110 register access: some
branch/condition check in the PROM boot code between reset and this loop is taking the wrong path
— either skipping the code that should arm the DCU decompress-to-WRAM step entirely, or jumping
into WRAM as if that step had already completed when it never started. Root-causing *which*
branch and *why* needs an instruction-level trace with real mnemonics (see next steps) — the
watchpoint hook narrowed *where* to look but can't show *what the CPU is deciding and why*
without decoding the actual instructions around `$00:F345-F416` and `$20:2000-2100`.

## `v0.8.0` update — a real 65C816 disassembler, a branch trace, and the exact derailment point

Built `rustysnes_cpu::disasm::disassemble_one` (`crates/rustysnes-cpu/src/disasm.rs`): a minimal,
decode-only disassembler for the standard WDC 65C816 opcode map (mnemonic + addressing mode +
operand-length table for all 256 opcodes, unit-tested against 11 hand-picked cases plus a
"every opcode decodes without panicking" sweep). Not wired into execution — pure debug-output
decode, reusable for the frontend's debugger overlay (`docs/frontend.md`) and any future
instruction-level trace, not SPC7110-specific.

Using it to single-step (`System::step_instruction`) from reset through the `$00:F416` loop this
file's earlier watchpoint pass found, and printing a disassembly window around every program-bank
transition, revealed the "`$00:F416`/`$20:20xx` stall loop" framing above was **itself
incomplete**: the CPU actually spends the vast majority of its time executing real, coherent code
in bank `$4F` — a legitimate VRAM-upload loop (`STA $2118`/`$2116`, the PPU's VRAM data/address
write ports), copying what is very likely title-screen/logo graphics into VRAM. The earlier
per-frame-granularity watchpoint sampling only ever caught the CPU at `$00:F416` because that's a
vblank-wait point between upload chunks — real, ongoing progress, not a stall, was happening
between samples.

That upload loop, however, does NOT run forever: it eventually reaches `C3:2B3A JSL $4FFB80` —
a real, literal `JSL` instruction (opcode `$22`, operand bytes `80 FB 4F`) confirmed present in
the raw `.sfc` dump at the corresponding PROM file offset (`0x032B3A`), not a read-path artifact.
Bank `$4F` is in the `$40-$7D` range. **Two real bugs were found and fixed here, cross-checked
directly against `ref-proj/ares/mia/Database/Super Famicom.bml`'s `board: SHVC-LDH3C-01` entry —
the exact board Far East of Eden Zero uses:**

- **The `$40-$7D` mcu-mapping range should be unmapped, not a `$C0-FF` mirror.** This project's
  `Spc7110Board::map`/`read24` treated `matches!(bank, 0x40..=0x7d | 0xc0..=0xff)` as one mcu-ROM
  region (an earlier session's undocumented, never-actually-checked claim that boot code
  "executes from this range"). The real board's `mcu` block declares exactly two `map` entries —
  `00-3f,80-bf:8000-ffff mask=0x800000` and `c0-ff:0000-ffff mask=0xc00000` — and `$40-$7D` does
  not satisfy either mask (bit 22 of the address differs). Fixed: both `map` and `read24` now
  route only `0xc0..=0xff` to `mcurom_read`; `$40-$7D` falls through to `MappedAddr::Open`/`0`
  (this project's open-bus stand-in), matching the real board exactly.
- **The DROM buffer was 2 MiB oversized.** The committed dump is exactly 7 MiB (`0x700000`), but
  the same `.bml` entry states the real physical chips are 1 MiB PROM + 4 MiB DROM = 5 MiB
  (`0x500000`) total — `select` was treating the trailing 2 MiB of the file (padding/junk some
  dump tools append) as real DROM, feeding `datarom_read`'s `bus_mirror` fold a 6 MiB length
  instead of the true 4 MiB. Fixed: `select` now slices exactly `PROM_SIZE=0x100000` +
  `DROM_SIZE=0x400000`, dropping anything past that.

Both fixes are independently correct (verified against the authoritative board database, not
guessed) and 9/9 `spc7110` unit tests plus the full workspace suite stay green. **Neither closes
the boot gap**: with `$40-$7D` now open bus, `JSL $4FFB80` lands on a constant-zero read (opcode
`$00` = `BRK`), producing the *same class* of BRK/RTI oscillation as before the fix, just
immediately and deterministically instead of after a few incoherent-looking instructions.

**The real open question, now precisely located:** the game's own PROM code (bank `$C3`, a real,
coherent VRAM-upload routine) contains a literal `JSL $4FFB80` — a jump into a bank the real
cartridge doesn't map to anything. A shipped, presumably-tested commercial title cannot
legitimately execute this on real hardware, which means **real hardware never actually reaches
`C3:2B3A` at all** — this emulation must diverge from real hardware *earlier* in the boot
sequence (somewhere before or during the VRAM-upload loop, or in whatever decides how many times
that loop's outer counter runs / what it does when it finishes) and this JSL is a downstream
symptom, not the disease. That earlier divergence has not yet been traced.

## `v0.8.0` update — a real, systemic open-bus bug found and fixed (still doesn't close the gap)

The "neither closes the boot gap" note above (`$40-$7D` now open bus, `JSL $4FFB80` lands on a
"constant-zero read") turned out to itself rest on an unverified assumption: **why would open bus
read back as a constant zero?** Checking ares' actual bus-read plumbing (`sfc/cpu/memory.cpp`,
`Bus::read`/`Bus::write` in `sfc/memory/inline.hpp`) shows real hardware's open bus does not
return zero — it returns the CPU's own MDR (Memory Data Register), the last byte value actually
driven onto the data bus, which the CPU itself threads through every bus access as the fallback
default (`bus.read(address, r.mdr)`; the "no handler mapped" reader echoes `data` straight back
unchanged). A `$00` opcode fetch from unmapped space is `BRK`; a real, non-zero echoed byte
usually is not — this distinction is the difference between "wild jump instantly traps" and "wild
jump keeps executing garbage-but-not-crashing code," a well-known real-hardware quirk many
commercial titles' crash-recovery and copy-protection code paths depend on.

This project's `Bus` already had the right *concept* — an `open_bus: u8` latch, doc-commented as
"the last value driven on the data bus," correctly consulted (without being overwritten) by every
other genuinely-unmapped bus region (register windows with no handler, the DMA A-bus's
CPU-register exclusion, etc, all already `_ => self.open_bus`). The cart layer was the one
exception: `Board::read24`'s trait-default and every board's own override (SPC7110's included)
returned a bare, hardcoded `0` for `MappedAddr::Open`/an unhandled range, and that `0` then got
latched into `Bus::open_bus` by the *following* `self.open_bus = val` in `CpuBus::read24` —
silently corrupting the latch with a wrong value instead of leaving it holding the real last-driven
byte. Boards have no access to the Bus's latch (correct layering — a board shouldn't know about
Bus-internal state), so they had no way to return anything else.

**Fix:** `Cart::read24` (`crates/rustysnes-cart/src/lib.rs`) now takes the caller's current
open-bus byte as a parameter, mirroring ares' `Bus::read(address, data)` exactly: if
`Board::map(addr24)` says the address is genuinely `MappedAddr::Open`, it returns the passed-in
open-bus byte directly, without ever calling into the board's own `read24` (whose hardcoded `0`
is only reachable now via direct unit tests that call a board's `read24` in isolation, not through
`Cart`/`Bus`). Both of `rustysnes-core`'s call sites (`Bus::cart_read_raw`, the DMA/CPU path; and
`CartView::cart_read`, the PPU's Mode 7/coprocessor split-borrow path) pass their own current
`open_bus` field through. No board-specific code changed — this fixes every board's cart-space
open bus at once, not just SPC7110's.

**Effect on this title, measured:** re-running the branch trace past `C3:2B3A` (rather than
stopping there) shows the CPU now fetches `$3D` repeatedly from bank `$4F` (the last real byte
latched — the high/bank byte of the `JSL $4FFB80` operand it had just fetched), decoding as
`AND $3D3D,X` — a harmless 3-byte read-modify instruction that free-runs across the entire bank,
wrapping `$4F:FFFF` back to `$4F:0000` and continuing forever. This is a **stable, non-crashing**
loop, confirming (independent of the board-database check already done) that bank `$4F` really is
inert on real hardware — nothing there can ever produce forward progress. It does **not** close
the boot gap: the CPU still never should have jumped there in the first place, so root cause #2 is
unchanged; this only makes today's specific dead end more honestly modeled (a stall, not a
spurious BRK-storm) while the real fix remains tracing backward from `C3:2B0E`.

## The resolution (`v0.8.0`, follow-up session) — the ROM dump was the fan-translation, not the original cart

Following the previous session's own suggested next step #1 ("trace backward from `C3:2B3A`... find
what decides `JSL $4FFB80` rather than continue/return normally") turned up something the framing
didn't anticipate: **there is no decision point to find.** A full instruction-level trace from
reset through the derailment (`crates/rustysnes-test-harness/tests/spc7110_trace.rs`, a throwaway
diagnostic built for this session and deleted afterward — the same `disassemble_one` +
`System::step_instruction` combination the prior session used) shows the entire path from the
first bank-`$C3` entry to `JSL $4FFB80` is one **unconditional** chain of `JSL`s inside a
straight-line init dispatcher at `$C0:2731-2745` (`JSL $C30000` → `JSL $C01DB5` → `JSL $C000E1` →
... → `JSL $C32B0E` → `JSL $C32B20` → return → `JSL $C32B3A` → `JSL $4FFB80`) — no `BNE`/`BEQ`/
branch/jump-table indirection anywhere in the 270,725 instructions leading up to it, and no
SPC7110 register (`$4800-483F`/`$4840-4842`) is ever touched in that entire span (re-confirmed,
matching the prior watchpoint trace). A wrong upstream branch was the leading hypothesis; it's
ruled out directly. That forced the question the prior sessions hadn't asked: **is this dump even
the right ROM?**

Three independent checks, all converging on the same answer:

1. **SHA256 mismatch against the authoritative database.** `ref-proj/ares/mia/Database/
   Super Famicom.bml`'s one and only entry for board `SHVC-LDH3C-01` (this exact board, used by
   no other game) records this title's canonical dump as `sha256:
   69d06a3f3a4f3ba769541fe94e92b42142e423e9f0924eab97865b2d826ec82d`, a 5 MiB total (1 MiB
   Program + 4 MiB Data, matching that entry's own `memory` nodes). The local dump is 7 MiB, and
   its SHA256 (of the full file, or of just the first 5 MiB, or with a 512-byte header stripped —
   every offset tried) never matches.
2. **The header checksum confirms the *dump* is internally consistent, but at the wrong size.**
   The header's checksum/complement pair (`$FFDE`/`$FFDC`, little-endian `$6A01`/`$95FE`) sums to
   exactly `$FFFF` — the standard "this header wasn't casually hand-edited" invariant — but the
   value `$6A01` is the actual byte-sum of the **entire 7 MiB file**, not the 5 MiB the canonical
   database says is the real cartridge's content. Whatever produced this file recalculated the
   header checksum for its own, larger, non-standard layout.
3. **A public, independent, primary source describes this exact patch's memory map.** A
   [nesdev.org forum thread on the Tengai Makyou Zero / Far East of Eden Zero English
   translation](https://forums.nesdev.org/viewtopic.php?t=18760) documents that the fan
   translation **restructures the ROM into three regions — 1 MiB Program, 5 MiB Data, and a new
   1 MiB "Expansion ROM"** (total 7 MiB, exactly this file's size) — because the translated script
   and re-compressed assets don't fit the original cart's fixed layout. Critically: **the
   Expansion ROM is mapped at CPU banks `$40-$4F`** — exactly the bank (`$4F`) this investigation
   traced the derailment into. The patch ships its own higan/bsnes mapping-definition file (a
   `HiganFileSplitter` tool divides the patched image into Program/Data/Expansion) precisely
   because this mapping **does not exist on any physical cartridge** and no stock SNES/Super
   Famicom board (including the real `SHVC-LDH3C-01`) implements it.

**Conclusion.** `JSL $4FFB80` is not a bug in this title's boot code, and RustySNES's mapping of
`$40-$7D` as unmapped (the real fix landed earlier in this same `v0.8.0` cycle, verified against
the authoritative board database) is **correct for the real cartridge** — it simply doesn't
implement the fan-translation's non-standard, patch-only "Expansion ROM" extension, which is not
part of any hardware this project models. Every one of this investigation's actual bug fixes
(root cause #1's `bus_mirror` addressing; the DCU/ALU trigger-timing deferral; the `$40-$7D`
mcu-mapping correction; the DROM-buffer-oversizing fix; the systemic cart-layer open-bus fix) are
genuine, independently-verified accuracy improvements that stand regardless of this finding — the
tooling and methodology were sound throughout. What was missing was checking the ROM's own
identity before spending further effort tracing its boot sequence as if it were the original
cartridge.

**This is now a ROM-sourcing question, not an emulation bug** — the same category as the
`docs/rom-test-corpus.md`-tracked PAL/ExLoROM/ST018/S-RTC real-ROM-validation gaps, not a
tracked-but-unresolved accuracy defect. `docs/STATUS.md`/`docs/cart.md`'s SPC7110 entries and
`docs/rom-test-corpus.md` are updated accordingly.

## What would actually close this out

Two independent paths, either sufficient on its own:

1. **Source a genuine, original-cartridge dump** matching SHA256
   `69d06a3f3a4f3ba769541fe94e92b42142e423e9f0924eab97865b2d826ec82d` (5 MiB, board
   `SHVC-LDH3C-01`) and re-run the exact same boot trace against it. If the CPU boots past this
   point cleanly (as the now-fixed `$40-$7D`/open-bus/DCU-timing/DROM-size bugs strongly suggest
   it should, since none of those fixes are specific to the fan-translation), SPC7110 moves from
   "BestEffort, not booting" to "BestEffort, real-title validated" — the same bar DSP-2/DSP-4/
   ST010/S-DD1/CX4/OBC1 already cleared.
2. **Or, as a deliberate, separate feature decision** (not implied by "fix the existing bug," and
   not attempted in this session without checking with the project owner first): add opt-in
   support for this fan-translation's documented `$40-$7D` (specifically `$40-$4F`) "Expansion
   ROM" mapping when a fourth ROM region of the right size is present. This would let RustySNES
   run this well-known, actively-distributed translation patch, at the cost of modeling
   patch-only behavior no real cartridge exhibits — a scope question for whoever picks this up,
   not a default-on accuracy fix.
