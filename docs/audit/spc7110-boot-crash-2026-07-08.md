# SPC7110 boot-crash investigation

**Status:** root cause #1 confirmed and fixed (`v0.4.0`); root cause #2 narrowed, not fixed —
tracked as future SPC7110 work per `docs/adr/0003`'s honesty gate (do not claim boot-validated
until it actually boots to real content).

**Board:** `crates/rustysnes-cart/src/coproc/spc7110.rs` (`coproc::spc7110`), `BestEffort` tier.
**Test title:** Far East of Eden Zero (Tengai Makyou Zero) — the one commercial SPC7110 title
with a local dump in this project's corpus. No other SPC7110 title has been tried.
**Spec entry:** `docs/cart.md`'s SPC7110 bullet (kept as the current-state summary; this file
is the full trail behind it).

## Background

SPC7110 is a Hudson ASIC combining a decompression unit (an adaptive binary range coder over
1/2/4bpp planes), a data-port unit, an ALU, and a memory-control unit that exposes four
independently-bankable 1 MiB data-ROM windows (`$4830`-`$4833` select, `$4834` selects the
window granularity). Unlike every other coprocessor in this project, SPC7110 carts physically
carry **two separate ROM chips** — a small PROM (CPU-executable program code, LoROM-style banks
`$00`-`$0F`) and a much larger DROM (compressed/plain data, addressed only through the board's
registers) — concatenated into one raw `.sfc` dump. There is no header field or generic formula
that recovers the PROM/DROM split for an arbitrary SPC7110 title; `coproc::spc7110::select`
guesses it from Far East of Eden Zero's documented physical geometry (1 MiB PROM + the
remainder as DROM, when the image is at least 2 MiB — see `select`'s doc comment for the
"no PROM" fallback used by titles like Momotarou Dentetsu VII, untested here).

## Root cause #1 — `bus_mirror` addressing (confirmed, fixed in `v0.4.0`)

### Symptom

Booting Far East of Eden Zero: the CPU ran off into unmapped low banks roughly 20-30 frames
into boot, BRK-storming (repeatedly trapping into the BRK vector, never recovering) per the
original diagnostic trace.

### Root cause

`datarom_read`/`mcurom_read`'s PROM/DROM lookups used a plain `offset % len` fold to keep a
register-selected address inside the physical chip's buffer. Real hardware (ares' `Bus::mirror`,
`sfc/memory/inline.hpp`) does something different: it repeatedly strips the largest power-of-two
block that keeps the address in range. The two algorithms agree only when the buffer size is
itself a power of two — and Far East of Eden Zero's DROM is not: `select` splits a 7 MiB image
into a 1 MiB PROM and a **6 MiB** DROM, and 6 MiB is not a power of two.

`$4830`-`$4833` can select up to an 8 MiB addressable window, larger than the physical 6 MiB
DROM. A register-selected read past the physical chip size but inside that addressable window
hit the `% len` fold and returned the **wrong byte** — silently corrupting whatever
data-ROM-resident table the game was reading through the data-port unit at the time.

### Fix

Ported the real `Bus::mirror` algorithm as `spc7110::bus_mirror` (`spc7110.rs:757`,
`const fn bus_mirror(address: u32, size: u32) -> u32`) and applied it to every PROM/DROM lookup
in `datarom_read` (`spc7110.rs:191`), `datarom_read_static` (`spc7110.rs:321`, the const-context
sibling used by the const-evaluable read paths), and `mcurom_read` (`spc7110.rs:432`).

### Effect (measured, not assumed)

The wild-PC excursion moved from ~20-30 frames into boot (permanent BRK-storm crash) to ~90+
frames, and the CPU now **self-recovers** via a BRK/RTI oscillation instead of crashing
permanently. This is real, measured forward progress — not a full fix — hence root cause #2
below.

## Root cause #2 — WRAM never populated before a real `RTI` targets it (open)

### Symptom (current, as of `v0.4.0`)

The CPU eventually executes a genuine `RTI` instruction, traced to `$CD:D2B3` — confirmed to be
real PROM-resident code, not decompression garbage or a wild-PC artifact. That `RTI` pops a
return address off the stack pointing into WRAM: `$20:0848` (mirror of `$7E:0848`). That WRAM
location is confirmed **entirely zeroed** at the point the `RTI` executes. The CPU then starts
"executing" zeroed WRAM as if it were code (`BRK` is opcode `$00`), traps into the BRK vector,
and the whole sequence oscillates — this is the "~90+ frames, self-recovering BRK/RTI
oscillation" behavior root cause #1's fix produced, not a full boot.

### Working hypothesis (not yet confirmed)

The game expects real code to already be resident at `$7E:0848` by the time this `RTI` fires —
very likely written there via a DCU-driven (decompression-unit-driven) decompress-to-WRAM step
earlier in the boot sequence that either (a) hasn't run yet when this `RTI` fires (an ordering
bug — this board's `coprocessor_tick`/data-port timing runs too slow or too fast relative to the
CPU's own progress), or (b) has run but is itself still buggy (a second, distinct data-path bug
in the decompression unit or the data-port write path that silently produces zeroed/wrong output
instead of the real decompressed code). Both are plausible from this board's implementation
maturity (`BestEffort` tier, no other data-path bug has been independently confirmed yet); this
has **not** been narrowed further than "one of these two."

### Why this hasn't been root-caused yet

Confirming which hypothesis (or a third, unconsidered one) is correct requires tracing exactly
what the DCU/data-port write sequence does to `$7E:0800`-`$7E:08FF` (or wherever the real write
target is) between reset and the crash — i.e. an instruction-level trace correlated against
every data-port register write, not just a PC trace. This project has no instruction-level
65C816 disassembler yet (`rustysnes-cpu` has a decoder for execution, not a standalone
disassembler-for-humans); building one is a real, reusable piece of debugging infrastructure
that would also help any future 65C816-side investigation, not just this one, but is out of
scope for a documentation-only session. Out of this session's scope — tracked here as future
SPC7110 work per `docs/adr/0003`'s honesty gate (not silently claimed fixed; `docs/cart.md` and
`docs/STATUS.md` both already say "not booting to real content" for this board).

## Suggested next steps (for whoever resumes this)

1. Build a minimal 65C816 disassembler (even a debug-only `Display` impl over decoded
   instructions would do) so the crash-point trace can show real mnemonics, not raw PC values.
2. Add a data-port write-trace hook (or a debug-only log line in `Spc7110Board`'s data-port
   register-write path) that records every byte written to WRAM via the DCU/data-port path,
   with the source PC, so the trace can be correlated against the `$7E:0848` zero-fill directly.
3. Re-run the Far East of Eden Zero boot with both traces active and check: does anything write
   to `$7E:0800`-`$7E:08FF` at all before the crashing `RTI`? If nothing does, this is an
   ordering bug (hypothesis a); if something does but the bytes are wrong, this is a data-path
   bug (hypothesis b) and the next step is comparing the written bytes against a real hardware
   trace or `ref-proj/ares`'s own SPC7110 core stepped against the same ROM.
4. Cross-reference `ref-proj/ares`'s `sfc/coprocessor/spc7110/` (decompression unit + data-port
   unit) for the exact trigger condition that starts a DCU decompress-to-WRAM run, since that is
   the most likely place an ordering or trigger-condition bug would hide.
